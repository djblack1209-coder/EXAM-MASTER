# EXAM-MASTER Full Project Audit Design

> 日期：2026-04-10
> 适用对象：本地全量审计、修复、验证与整理
> 当前边界：先本地后远程；本轮不直接推送远程仓库、不直接登录服务器

## 1. 背景

本项目已经经历多轮集中修复，当前 `docs/status/HEALTH.md` 显示主体系统可运行，但仍存在以下现实问题：

1. 仍有最高级安全风险未完全闭环，例如 `H024` 历史环境变量明文暴露后的轮换与收口问题。
2. 已修复项很多，但部分修复属于“点状修补”，缺少一次从生产风险到长期维护的一体化复盘。
3. 项目体量较大，包含 uni-app 前端、Laf 云函数、独立后端、Electron、构建脚本、CI/CD、视觉回归、E2E、文档体系，任何单点遗漏都可能在真实用户使用时暴露。
4. 用户要求的不只是代码层审查，而是覆盖 UI、UX、后端、架构、运维、API、文件治理、构建与桌面入口可用性的全项目经营级审计。

因此，本轮工作不是“挑几个 bug 修掉”，而是要按价值位阶重新梳理项目，优先清除生产级风险，再补齐功能完整性和体验一致性，最后完成文档与治理收口。

## 2. 目标

本轮目标是完成一次“价值位阶驱动”的全项目审计与修复，使 EXAM-MASTER 在本地达到以下状态：

1. 关键安全、鉴权、构建、数据正确性、核心功能链路不存在明显生产阻断。
2. 所有用户可见区域都经过截图或可视化验证，重点覆盖深浅色、空态、弹窗、加载、排版、对齐、AI 输出渲染。
3. 所有用户可能触发的操作都尽量经过模拟点击、输入、切换、提交流程验证，重点识别失效按钮、假交互、mock 数据、接口未接通、异常提示缺失。
4. 识别并收敛分层违规、死代码、重复实现、脏文件、断裂资源、文档颗粒度不一致等长期维护风险。
5. 完成本地发布级验证，包括 lint、单测、H5 构建、小程序构建、后端 TypeScript 编译、Electron 启动核验，以及按改动等级触发的 E2E / 可视化回归。
6. 在文档层留下清晰证据：审计范围、修复结果、已知剩余问题、下一轮动作。

## 3. 非目标

本轮默认不做以下事项，除非用户后续明确要求：

1. 不直接推送远程仓库。
2. 不直接连接腾讯云服务器执行部署、PM2、Nginx、MongoDB、SSL 实机操作。
3. 不为“理论上可能更优”而做大规模无关重构。
4. 不因为追求整洁而破坏现有可运行行为。

## 4. 设计原则

### 4.1 价值位阶优先

处理顺序固定为：

1. 生产风险
2. 核心功能完整性
3. 用户界面与交互体验
4. 架构治理与文件治理
5. 构建发布与桌面入口
6. 文档对齐与审计留痕

这意味着任何“看起来不美观但不影响生产”的问题，都排在安全、数据正确性、构建阻断和核心链路之后。

### 4.2 先全局扫描，再分桶修复

不采用“看到一个问题修一个”的零碎方式，而是先建立问题地图：

1. 安全与配置桶
2. 构建与发布桶
3. 核心链路桶
4. UI/UX 桶
5. 架构与文件治理桶
6. 文档与维护桶

这样可以避免来回返工，也能保证每次改动后的回归验证有边界。

### 4.3 证据优先

所有“正常”“可用”“修复完成”的结论，都必须来自以下证据之一：

1. 命令行验证通过
2. Playwright 截图或交互结果
3. 单元测试 / E2E / 可视化回归结果
4. 代码路径核查与实际行为一致

### 4.4 小步快跑，按桶收口

每个审计桶内部采用“小范围扫描 -> 修复 -> 验证 -> 记录”的闭环，不把所有问题堆到最后一起验证。

## 5. 审计范围

### 5.1 P0 生产阻断层

目标：优先消灭会伤用户、炸生产、挡上线的问题。

范围包括：

1. 密钥、凭据、环境变量、签名校验、鉴权中间件、敏感文件入库风险。
2. 前端是否绕过 Service 层直连后端。
3. 后端需要登录的接口是否缺少 `requireAuth(ctx)` 或等效鉴权。
4. AI 代理链路、流式输出链路、签名头要求、内容安全检查。
5. 构建阻断：H5、小程序、Electron、后端 TS 编译。
6. 假数据或 mock 数据是否误流入生产路径。
7. Electron 外链、桌面入口、启动白屏等高风险问题。

### 5.2 P1 核心功能完整性层

目标：保证用户最常用、最关键的链路真的能走通。

重点链路：

1. 启动页、登录页、首页、四个 Tab 页面。
2. 刷题链路：题库选择、做题、提交、结果、FSRS 调度、智能复习。
3. AI 链路：AI 聊天、AI 课堂、择校 AI 咨询。
4. 数据链路：错题本、收藏、学习计划、导入导出、文件管理。
5. 社交链路：PK、排行榜、好友列表、好友主页。
6. 工具链路：拍照搜题、文档转换、证件照、专注计时。
7. 设置链路：主题切换、隐私协议、AI 家教列表、分享与邀请。

### 5.3 P2 全量 UI / UX 层

目标：把所有用户看得到、点得到、切得到的地方全部过一遍。

验证维度：

1. 页面截图：主页面、功能页面、关键弹窗、空态、加载态、错误态。
2. 深色模式 / 浅色模式双模对比。
3. 排版：字间距、行高、清晰度、是否溢出、是否裁切。
4. 布局：容器漂移、遮挡、TabBar 遮盖、安全区问题。
5. 交互：按钮可点击、禁用态是否有反馈、弹窗能否关闭、加载是否结束。
6. AI 输出：Markdown、列表、代码块、公式、流式内容、思考中状态。

### 5.4 P3 架构治理 / 文件治理层

目标：把“现在看起来还能跑，但以后一定难维护”的问题提前处理掉。

范围包括：

1. 分层违规：Page / Component / Store / Service / Backend 越层调用。
2. 大文件高风险区：`do-quiz.vue`、`pk-battle.vue`、`index.vue`、`school/index.vue`、`settings/index.vue` 等超大文件。
3. 死代码、重复实现、未引用资源、断裂软链接、临时截图、错误入库文件。
4. 配置与目录颗粒度不一致：脚本、构建配置、测试配置、文档索引是否对齐。
5. 文档与现实不一致：统计数字、流程说明、部署现状、测试基线。

### 5.5 P4 发布 / 兼容 / 桌面入口层

目标：保证本地达到“可发布前夜”的状态。

范围包括：

1. `npm run lint`
2. `npm test`
3. `npm run build:h5`
4. `npm run build:mp-weixin`
5. `laf-backend` TypeScript 编译检查
6. Electron 启动核验
7. 改动触发后的 E2E / 可视化回归 / 全量回归选择

### 5.6 P5 文档与审计留痕层

目标：修完后不是留下一地碎片，而是形成可接手、可复盘的项目状态。

输出包括：

1. `docs/sop/CHANGE-LOG.md` 更新
2. `docs/status/HEALTH.md` 更新
3. 必要时补充审计结果文档或模块说明文档
4. 明确本轮完成项、未完成项、后续远程步骤

## 6. 执行策略

### 6.1 第一步：建立问题地图

执行以下动作：

1. 读取当前代码结构、测试入口、构建脚本、页面配置、Electron 入口、后端函数目录。
2. 跑第一轮无侵入检查：工作区状态、脚本清单、关键文件搜索、构建配置与已知高风险点搜索。
3. 将发现的问题按严重程度分类为：
   - S0：安全 / 数据 / 构建阻断 / 崩溃
   - S1：核心功能不完整或伪可用
   - S2：明显 UI/UX 问题或交互断点
   - S3：架构治理与维护风险

### 6.2 第二步：按桶修复

修复顺序如下：

1. 先修 S0
2. 再修 S1
3. 然后做 S2 截图比对与交互修补
4. 最后处理 S3 整理与文档同步

每一桶必须在退出前完成对应验证，不允许把未验证代码继续叠到下一桶。

### 6.3 第三步：高价值开源复用判断

如果在审计过程中发现以下情况，可以引入成熟方案，但必须满足“最小接入”原则：

1. 当前实现明显不稳定，且已有高 Star、可裁剪的成熟方案。
2. 引入后不会破坏现有分层，也不会引入重型依赖导致小程序主包超限。
3. 有明确收益，例如更稳定的 Markdown 渲染、截图审计工具、配置治理脚本、静态检查规则。

如果只是“别人实现更漂亮”，但当前问题并不成立，则不引入。

## 7. 关键检查项

### 7.1 安全与配置

1. 搜索真实密钥、密码、RSA 私钥、公网凭据误入库。
2. 搜索 `.env`、Worker、CI、前端常量中的敏感字段暴露。
3. 搜索 `requireAuth(ctx)` 缺失场景。
4. 搜索 mock 数据、占位接口、假成功返回。

### 7.2 API 与数据正确性

1. 检查 Service 层是否完整封装，避免页面或 Store 越层。
2. 检查前后端响应结构是否一致。
3. 检查错误提示是否到用户可见层，而不是只打日志。

### 7.3 UI / UX

1. 跑主页面与高风险页面截图。
2. 逐页查看深浅色切换。
3. 覆盖弹窗、空态、加载、列表长内容、AI 富文本输出。
4. 对关键流程做真实点击和输入。

### 7.4 构建 / 兼容 / 桌面

1. H5 构建与小程序构建必须可用。
2. Electron 入口必须能打开，不白屏，不因协议或资源路径失败。
3. 后端编译必须通过，避免“前端看似正常、部署即炸”。

## 8. 验证策略

### 8.1 基线验证

在开始大范围修复前，先跑并记录基础状态：

1. `npm run lint`
2. `npm test`
3. `npm run build:h5`

### 8.2 按改动触发扩展验证

根据 `docs/sop/REGRESSION-TEST-STRATEGY.md`：

1. 改页面逻辑或交互，补 E2E。
2. 改样式或主题，补可视化回归。
3. 改多个组件、后端、构建配置或大规模重构，补全量回归或尽量接近全量回归。
4. 改后端时补 `laf-backend` TypeScript 编译。
5. 改跨平台入口时补小程序构建和 Electron 启动核验。

### 8.3 证据产出

最终需要保留以下证据中的尽可能多项：

1. 命令输出结果
2. 截图文件或可视化回归结果
3. 关键页面/链路的通过记录
4. 文档更新记录

## 9. 风险与约束

1. 项目页面和组件数量较大，本轮可能无法在一次对话内完成所有桶的彻底收口。
2. 部分功能依赖真实后端、真实账号、外部 AI 供应商或平台环境，纯本地验证可能只能确认“前端链路完整 + 请求路径正确 + 无明显假数据”。
3. 本轮不直连服务器，因此无法闭环服务器运行态、PM2、Nginx、SSL 与真实线上健康检查，只能先做好本地发布前核验。
4. 历史脏工作区或并行改动可能存在，必须避免误覆盖非本轮修改。

## 10. 交付物

本次设计落地后的交付应包括：

1. 一份实施计划文档，拆到可执行阶段。
2. 一轮实际审计与修复结果。
3. 更新后的 `CHANGE-LOG.md` 与 `HEALTH.md`。
4. 若本轮无法彻底完成，则输出“已完成 / 未完成 / 建议下一轮”的明确边界。

## 11. 推荐执行顺序

1. 基线扫描与问题地图
2. P0 生产阻断层
3. P1 核心功能完整性层
4. P2 全量 UI / UX 层
5. P3 架构治理 / 文件治理层
6. P4 发布 / 兼容 / 桌面入口层
7. P5 文档与审计留痕层

## 12. 成功标准

本轮可以被认定为“阶段完成”，需要同时满足：

1. 发现的问题已经按严重级别完成扫描与分类。
2. S0 问题全部修复或明确隔离，没有悬空的生产阻断。
3. S1 问题完成最大可能范围的修复与验证。
4. 用户可见界面完成高覆盖截图与关键交互验证。
5. 本地质量关卡通过，或未通过项已被定位并列为唯一剩余阻塞项。
6. 文档已同步，后续动作清晰。

## 13. 当前建议

按照本设计继续推进，先写实施计划，再正式进入执行。执行中如发现某一桶规模过大，则允许在桶内部进一步分批，但不改变总顺序。
