## 二、P1 严重问题

### P1-01: 好友请求未做重复发送的竞态防护（后端）
- **文件**: `laf-backend/functions/social-service.ts:189-201`
- **维度**: 数据完整性
- **描述**: `handleSendRequest` 中先查询是否已存在好友关系，再插入新记录，但查询与插入之间无原子性保证。并发场景下两个用户可同时通过检查，导致 `friends` 集合中出现重复的 pending 记录。
- **影响**: 好友关系表出现脏数据，用户可能收到重复的好友请求，后续 `handleHandleRequest` 的 `getOne()` 可能命中错误记录。
- **修复建议**: 在 `friends` 集合上为 `(user_id, friend_id)` 建立唯一索引，或使用 `upsert` 操作替代先查后写模式，在数据库层面保证唯一性。

### P1-02: 搜索接口读操作未强制鉴权，可被匿名滥用
- **文件**: `laf-backend/functions/social-service.ts:46-63`
- **维度**: 安全
- **描述**: `search_user` 和 `get_friend_list`、`get_friend_requests` 等读操作不在 `writeActions` 列表中，当请求不携带 token 时 `userId` 可由客户端 `params.userId` 任意传入（第43行），服务端不做验证即信任该值。
- **影响**: 攻击者可伪造任意 `userId` 调用 `get_friend_list` 获取他人好友列表及个人信息，造成用户隐私泄露。
- **修复建议**: 将所有需要 `userId` 的操作（包括 `get_friend_list`、`get_friend_requests`）加入强制 JWT 验证列表，或至少在读操作中当 token 缺失时拒绝使用客户端传入的 `userId`。

### P1-03: 证件照 Base64 图片数据直接存储在内存，无大小限制校验
- **文件**: `src/pages/tools/id-photo.vue:231-261`
- **维度**: 安全 / 性能
- **描述**: `readBase64` 方法将用户选择的照片完整转为 Base64 字符串存入 `this.imageBase64`，未对图片文件大小做任何前端校验。超大图片（如 20MB+ 的高分辨率照片）会导致 Base64 字符串占用约 27MB 内存，并在网络请求时传输巨量数据。
- **影响**: 小程序端可能因内存不足崩溃；后端接收超大 payload 可能超时或被 WAF 拦截；用户体验极差。
- **修复建议**: 在 `choosePhoto` 成功回调中检查 `res.tempFiles[0].size`，限制最大 5MB；超限时提示用户压缩或更换照片。同时在 `chooseImage` 中使用 `compressed` 模式（已有）并考虑增加 `canvas` 压缩步骤。

### P1-04: 文档转换轮询定时器在页面异常退出时可能泄漏
- **文件**: `src/pages/tools/doc-convert.vue:297-327`
- **维度**: 性能 / 功能完整性
- **描述**: `startPolling` 使用 `setInterval` 每 2 秒轮询后端状态，虽然 `onUnload` 中调用了 `clearPollTimer`，但如果用户通过系统手势返回（部分安卓机型不触发 `onUnload`）或小程序被系统回收，定时器不会被清除，持续发起无效网络请求。此外轮询期间无网络错误累计退出机制，连续失败会无限重试直到 `maxAttempts`。
- **影响**: 后台持续发起无效请求，消耗用户流量和服务端资源；连续网络错误时用户无感知。
- **修复建议**: 增加连续错误计数器，超过 3 次连续失败时停止轮询并提示用户；在 `onHide` 生命周期中也暂停轮询，`onShow` 时恢复。

### P1-05: 好友资料页通过 URL 参数传递完整用户数据，存在信息篡改风险
- **文件**: `src/pages/social/friend-list.vue:694-708` / `src/pages/social/friend-profile.vue:258-279`
- **维度**: 安全 / 数据完整性
- **描述**: `goToFriendProfile` 将好友的 `uid`、`nickname`、`avatar`、`score`、`studyDays`、`accuracy` 等全部数据通过 URL query 参数传递，`friend-profile.vue` 直接从 `options` 中读取并展示，不做任何后端校验。
- **影响**: 用户可通过修改 URL 参数伪造任意好友资料页面（如篡改分数、昵称）；URL 中包含完整头像地址等敏感信息，可能被日志系统记录泄露。
- **修复建议**: 仅传递 `uid`，在 `friend-profile.vue` 的 `onLoad` 中通过后端接口根据 `uid` 拉取好友详细信息。

### P1-06: 文档转换后端未对 fileBase64 做格式校验，可能被注入恶意文件
- **文件**: `laf-backend/functions/doc-convert.ts:144-206`
- **维度**: 安全
- **描述**: `handleConvert` 仅检查了 `fileBase64` 是否存在和文件大小，但未校验 Base64 内容的合法性（是否为有效 Base64 编码），也未校验文件扩展名是否与 `convertType` 声明的输入格式匹配。攻击者可提交任意 Base64 数据（如可执行文件）并声明为 `word2pdf` 类型。
- **影响**: 恶意文件可能被传递给 CloudConvert API 处理，触发安全漏洞或消耗 API 配额。
- **修复建议**: 校验 `fileName` 的扩展名是否在 `CONFIG.supportedFormats[convertType].input` 列表中；对 Base64 数据做正则格式校验 `/^[A-Za-z0-9+/=]+$/`；可选增加文件魔数（magic bytes）校验。

## 三、P2 中等问题

### P2-01: socialService.js 搜索结果数据结构不兼容后端分页格式
- **文件**: `src/pages/social/socialService.js:15-29` / `src/pages/social/friend-list.vue:481-482`
- **维度**: 功能完整性
- **描述**: 后端 `search_user` 返回 `{ data: { list, total, page, hasMore } }` 分页结构，但前端 `socialService.searchUser` 直接透传 `res`，而 `friend-list.vue:482` 使用 `res.data` 作为数组赋值给 `searchResults`。实际 `res.data` 是一个包含 `list` 字段的对象，不是数组。
- **影响**: 搜索结果无法正确渲染，`v-for="user in searchResults"` 遍历的是对象属性而非用户列表。
- **修复建议**: 在 `friend-list.vue:482` 中改为 `this.searchResults = res.data?.list || res.data || []`，或在 `socialService.searchUser` 中统一返回格式。

### P2-02: 好友列表 isAddingFriend 响应式对象动态属性不触发视图更新
- **文件**: `src/pages/social/friend-list.vue:305-306,562,591`
- **维度**: 功能完整性
- **描述**: `isAddingFriend`、`isAccepting`、`isRejecting` 在 `data()` 中初始化为空对象 `{}`，后续通过 `this.isAddingFriend[user._id] = true` 动态添加属性。在 Vue 2 Options API 中，动态添加的对象属性不是响应式的，视图不会更新。
- **影响**: 按钮的 `:disabled` 和文本切换（如"发送中..."）不会生效，用户可重复点击触发多次请求。
- **修复建议**: 使用 `this.$set(this.isAddingFriend, user._id, true)` 替代直接赋值，或改用 `Vue.set`。

### P2-03: 好友列表 onShow 首次加载逻辑存在标志位竞态
- **文件**: `src/pages/social/friend-list.vue:333-351`
- **维度**: 功能完整性
- **描述**: `onLoad` 中设置 `this._loadedOnce = true`，`onShow` 中检查 `!this._loadedOnce` 来避免重复加载，然后立即设为 `false`。但 `_loadedOnce` 是挂在 `this` 上的非响应式属性，且逻辑上 `onShow` 在 `onLoad` 之后立即触发时，`_loadedOnce` 为 `true` 会跳过加载——这是预期行为。但当页面从后台恢复时 `_loadedOnce` 已被设为 `false`，会正确刷新。问题在于如果 `loadFriendList` 是异步的，`onShow` 可能在 `onLoad` 的异步请求完成前就执行，导致短时间内发起两次请求。
- **影响**: 页面首次加载时可能发起重复的网络请求，浪费资源。
- **修复建议**: 使用一个 `isLoading` 标志位来防止并发请求，或在 `onShow` 中检查 `isLoading` 状态。

### P2-04: 证件照处理中步骤无超时保护和取消机制
- **文件**: `src/pages/tools/id-photo.vue:263-297`
- **维度**: 功能完整性
- **描述**: `startProcess` 调用 `lafService.processIdPhoto` 后等待后端返回，但无超时控制。如果后端处理时间过长（如 AI 抠图服务响应慢），用户会一直停留在"处理中"界面，无法取消或返回。
- **影响**: 用户体验差，可能误以为应用卡死；无法中断已发起的请求。
- **修复建议**: 使用 `Promise.race` 配合超时 Promise（如 30 秒），超时后提示用户并回退到 step 1；在处理中界面增加"取消"按钮。

### P2-05: 文档转换前端未校验文件扩展名与所选转换类型是否匹配
- **文件**: `src/pages/tools/doc-convert.vue:204-241`
- **维度**: 数据完整性
- **描述**: `chooseFile` 方法在微信小程序端使用 `wx.chooseMessageFile` 选择文件，但未校验所选文件的扩展名是否与当前 `selectedType` 对应的 `accept` 格式匹配。用户可能选择了 `word2pdf` 类型后上传一个 `.xlsx` 文件。
- **影响**: 文件提交到后端后转换失败，浪费用户等待时间和 API 配额。
- **修复建议**: 在文件选择成功后，提取文件扩展名并与 `this.currentType.accept` 列表比对，不匹配时提示用户重新选择。

### P2-06: 后端删除好友未验证好友关系是否存在
- **文件**: `laf-backend/functions/social-service.ts:411-437`
- **维度**: 数据完整性
- **描述**: `handleRemoveFriend` 直接执行 `remove()` 操作，未先检查好友关系是否存在。如果传入的 `friendUid` 不是当前用户的好友，`remove()` 返回删除 0 条记录但仍返回成功。
- **影响**: 前端收到成功响应后执行页面返回和缓存清除，但实际未删除任何数据，用户可能困惑。
- **修复建议**: 在 `remove()` 后检查 `deleted` 数量，为 0 时返回 `{ code: 404, message: '好友关系不存在' }`。

### P2-07: 证件照 H5 端 saveResult 使用 `link.click()` 下载不兼容部分浏览器
- **文件**: `src/pages/tools/id-photo.vue:340-345`
- **维度**: 功能完整性
- **描述**: H5 端保存结果使用动态创建 `<a>` 标签并设置 `download` 属性后 `click()`，但 `download` 属性在 iOS Safari 中对 `data:` URL 不生效，且部分浏览器会直接打开图片而非下载。
- **影响**: iOS H5 用户无法保存证件照结果。
- **修复建议**: 将 Base64 转为 Blob URL 后再触发下载，或使用 `canvas.toBlob()` + `URL.createObjectURL()` 方案；对不支持的浏览器提示长按保存。

### P2-08: 文档转换后端 waitForJobComplete 同步阻塞云函数执行
- **文件**: `laf-backend/functions/doc-convert.ts:282-358`
- **维度**: 性能
- **描述**: `handleConvert` 调用 `waitForJobComplete` 在云函数内部以 while 循环 + sleep 方式同步等待最长 120 秒。Laf/Sealos 云函数通常有执行时间限制（60-120秒），长时间占用会导致函数超时或占满并发槽位。
- **影响**: 大文件转换时云函数可能超时返回错误；高并发时阻塞其他请求。
- **修复建议**: 改为异步模式：`handleConvert` 仅创建任务并立即返回 `taskId`，由前端轮询 `get_status` 接口查询进度（前端已有轮询逻辑但未被使用）。

### P2-09: 好友资料页成就数据为硬编码，未从后端获取
- **文件**: `src/pages/social/friend-profile.vue:227-233`
- **维度**: 功能完整性
- **描述**: `achievements` 数组在 `data()` 中硬编码为固定的三个成就 `['连续7天', '刷题达人', '精准射手']`，与好友的实际数据无关。所有好友资料页显示完全相同的成就。
- **影响**: 用户看到的成就信息不真实，降低社交功能的可信度和价值。
- **修复建议**: 从后端接口获取好友的实际成就数据，或根据好友的 `studyDays`、`accuracy`、`score` 等字段动态计算成就解锁状态。

## 四、P3 优化建议

### P3-01: socialService.js 中 console.error 与 logger 混用
- **文件**: `src/pages/social/socialService.js:27,51,75,120,177`
- **维度**: 代码质量
- **描述**: 文件中同时使用 `logger.log` 和 `console.error`，catch 块中统一使用了 `console.error` 而非 `logger.error`。项目已引入统一日志工具 `logger`，生产环境下 `console.error` 不受日志级别控制。
- **影响**: 生产环境日志输出不一致，可能泄露错误堆栈信息。
- **修复建议**: 将所有 `console.error` 替换为 `logger.error`，保持日志工具统一。

### P3-02: friend-list.vue 和 friend-profile.vue 中时间格式化逻辑重复
- **文件**: `src/pages/social/friend-list.vue:738-776` / `src/pages/social/friend-profile.vue:237-256`
- **维度**: 代码质量
- **描述**: `isOnline`、`getStatusText`/`statusText`、`formatTime` 等时间相关逻辑在两个文件中几乎完全重复实现。
- **影响**: 维护成本高，修改逻辑需同步两处。
- **修复建议**: 提取为公共 mixin 或 composable（如 `useFriendStatus.js`），两个页面共享。

### P3-03: 证件照页面 changeColor 仅回退步骤，未保留抠图结果
- **文件**: `src/pages/tools/id-photo.vue:300-302`
- **维度**: 功能完整性
- **描述**: `changeColor` 方法仅将 `step` 设为 1 回到配置页，用户更换颜色后点击"开始制作"会重新上传原图并完整执行抠图流程。已保存的 `foregroundBase64`（前景抠图结果）未被复用。
- **影响**: 用户仅想换背景色却需等待完整的 AI 处理流程，体验不佳且浪费 API 调用。
- **修复建议**: 如果 `foregroundBase64` 已存在，换色时仅调用合成接口（前景+新背景色），跳过抠图步骤。

### P3-04: 文档转换后端日志输出 Base64 数据的前 200 字符
- **文件**: `laf-backend/functions/doc-convert.ts:52`
- **维度**: 安全
- **描述**: 第 52 行 `console.log` 输出 `ctx.body` 的前 200 字符，当 body 中包含 `fileBase64` 时会将文件内容片段写入日志。
- **影响**: 日志中可能包含用户文件的部分内容，存在数据泄露风险。
- **修复建议**: 在日志输出前排除 `fileBase64` 字段，或仅输出 `action`、`convertType`、`fileName` 等元数据。

### P3-05: 后端搜索用户执行了两次相同的 where 查询（count + get）
- **文件**: `laf-backend/functions/social-service.ts:138-147`
- **维度**: 性能
- **描述**: `handleSearchUser` 使用 `Promise.all` 同时执行 `count()` 和分页 `get()` 查询，两次查询的 `where` 条件完全相同，数据库需要执行两次索引扫描。
- **影响**: 搜索请求的数据库负载翻倍，高并发时影响响应速度。
- **修复建议**: 如果数据库驱动支持，使用单次聚合查询同时返回总数和分页数据；或先执行 `get()` 获取 `pageSize + 1` 条记录来判断 `hasMore`，省去 `count()` 调用。
